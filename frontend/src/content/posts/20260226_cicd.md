# CI/CD 介紹

CI/CD 是現代軟體開發中非常核心的自動化流程，全稱是 持續整合（Continuous Integration） 與 持續佈署（Continuous Deployment/Delivery）。

簡單來說，它的目標是透過自動化手段，確保程式碼在進入正式環境前，就已經過嚴格的檢查，
系統像個嚴格的守門員，先幫你跑過一遍所有的檢查，確認沒問題了才准許自動部屬，避免「手動佈署」導致的人為錯誤。

## CI：持續整合 (Continuous Integration)
- 這是開發階段的自動化。
  程式碼推送到 GitHub 或 GitLab 時，系統會自動執行以下動作：
    - 自動編譯：檢查程式碼是否能順利跑起來。
    - 代碼檢查 (Linting)：確保程式碼風格符合規範（例如使用 ESLint）。
    - 自動測試：執行單元測試，確保新功能沒有壞掉舊有的邏輯。
    - 型別檢查：如果是使用 TypeScript，會檢查是否有型別錯誤。
    - 核心價值： 在問題進入主分支前就先抓出來，達成「提早發現問題，提早修復」。


## CD：持續交付/佈署 (Continuous Delivery / Deployment)
- 這是發佈階段的自動化「自動部屬」。
- CD (Continuous Deployment) 的核心
    - 持續交付 (Continuous Delivery)：自動準備好可以佈署的版本（Artifact），但最後一棒「上線」仍由人工點擊確認。
    - 持續佈署 (Continuous Deployment)：只要前面的測試全部通過，系統就自動將程式碼推送到伺服器（例如 Vercel、Supabase 或 Docker 環境），完全不需要人工干預。


## 常見的 CI/CD 工具組合
- GitHub Actions：目前最主流，直接整合在 GitHub 倉庫中。
- GitLab CI/CD：內建在 GitLab 裡，功能非常強大。
- Jenkins：老牌、自由度最高，但維護成本較重。


## CI/CD 的完整流程
1. **提交程式碼 (Git Push)**：你把改好的功能推送到 GitHub。
2. **持續整合 (CI) - 自動檢查：**
    - 系統自動啟動一台虛擬主機。
    - 執行 Linting（檢查程式碼格式）。
    - 執行 Type Check（檢查型別是否有誤，例如 TypeScript）。
    - 執行 Unit Test（跑測試案例，確保功能邏輯正確）。
    - 如果這一步失敗，流水線會直接中斷，不會進行部屬。 這就是為了達成「在部屬前就發現問題」。
3. **持續部屬 (CD) - 自動上線：**
    - 如果檢查全部通過，系統會自動將程式碼打包。
    - 自動推送到雲端平台（如 Supabase、Vercel 或 Azure）。
    - 完成更新，使用者立刻看到新功能。
